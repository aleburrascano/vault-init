# Domain Language

- **vault** = a local Obsidian directory containing `.obsidian/`, `CLAUDE.md`, `raw/`, and `wiki/` subdirectories, registered in `~/.claude.json` for Claude Code MCP access.
- **vault name** = user-chosen identifier matching `^[a-zA-Z0-9_-]+$`, max 64 chars; used as the key in the MCP registry.
- **vault dir** = the full filesystem path to a vault on disk; always resolved via the MCP registry, never from user input.
- **MCP registry** = the `mcpServers` object in `~/.claude.json`, where each vault is registered with its path and expected SHA-256 hash.
- **launcher** = `.mcp-start.js` in each vault root; bytes copied verbatim from `lib/mcp-start.js.tmpl`, pinned to a SHA-256 hash for self-verification.
- **launcher template** = `lib/mcp-start.js.tmpl` — the single source of truth for the launcher. Stays as raw JS (not migrated to TS) because every user vault byte-pins its SHA-256.
- **GitHub Pages** = static site hosting integrated via `src/commands/visibility.ts` and `lib/deploy.yml.tmpl` for publishing `raw/` and `wiki/` content.
- **dispatch** = the flow `vaultkit <cmd>` → `bin/vaultkit.ts` (commander) → `src/commands/<cmd>.ts`. The published package ships compiled `dist/bin/vaultkit.js` → `dist/src/commands/<cmd>.js`.
- **command module** = a `src/commands/<name>.ts` file that exports `async function run(params, options?: <Name>Options): Promise<...>`. Per-command `<Name>Options` interfaces extend `RunOptions` from `src/types.ts`.
- **lib module** = a `src/lib/<name>.ts` file shared by command modules. Organized into five domain clusters under `src/lib/`: `github/` (`gh-retry`, `github-repo`, `github-pages`, `github-auth`), `search/` (`search-index`, `search-indexer`, `sqlite-backend`, `vault-walk`), `mcp/` (`mcp`, `mcp-stdio`, `json-rpc-validator`), `templates/` (`vault-templates`, `vault-layout`, `template-paths`, `claude-md-merge`), `freshness/` (`sources`, `report`). Cross-cutting libs at the `src/lib/` root: `registry`, `vault`, `platform`, `prereqs`, `git`, `poll`, `errors`, `logger`, `messages`, `constants`, `update-check`, `text-compare`, `launcher-history`, `post-upgrade-check`, `preflight-launcher`, `breaking-changes`. Plus the per-tool implementations under `src/mcp-tools/`.
- **setup** = the `vaultkit setup` command (`src/commands/setup.ts`) — one-time post-install onboarding that walks the user through every prerequisite (node, git, gh + `repo,workflow` scopes, git config, claude). Idempotent. Returns the count of unresolved issues. Output mirrors `doctor`'s `+ ok` / `! warn` / `x fail` vocabulary; the difference is `doctor` reports while `setup` actively fixes. Per the security invariant, setup never requests `delete_repo`. Also performs one-time legacy cleanup (per ADR-0011): if the pre-2.8 `vaultkit-search` MCP entry exists, removes it; if `~/.vaultkit/search-launcher.js` exists, deletes it. `~/.vaultkit-search.db` is preserved (per-vault MCP servers still read it).
- **vaultkit MCP server** = the per-vault MCP server (`vaultkit mcp-server` subcommand, `src/commands/mcp-server.ts`) spawned by the byte-immutable per-vault launcher on every Claude Code session. Replaces the previous shape (per-vault `obsidian-mcp-pro` plus a separate global `vaultkit-search` MCP) per ADR-0011. Hand-rolls JSON-RPC 2.0 over stdio (no `@modelcontextprotocol/sdk` dependency); registers six tools tuned for Claude (`vk_search`, `vk_list_notes`, `vk_get_note`, `vk_get_tags`, `vk_search_by_tag`, `vk_recent_notes`). Reads the shared `~/.vaultkit-search.db` (SQLite FTS5 + BM25, via `node:sqlite` — Node ≥22.13). Each tool defaults to scoping by the launcher's vault; pass `vault: "*"` for cross-vault search across every registered vault, or a specific vault name to scope to one. The legacy `vaultkit-search` MCP entry was retired in 2.8; existing users migrate via `vaultkit setup` (cleans up legacy state) plus `vaultkit update <name>` per vault (re-pins the launcher to the new SHA — the new launcher spawns `vaultkit mcp-server` instead of `npx obsidian-mcp-pro`).
- **vk_* tools** = the six MCP tools implemented under `src/mcp-tools/<vk-name>.ts`. Each file exports a `<vkName>Definition(ctx: ToolContext): ToolDefinition` factory; `src/mcp-tools/index.ts` builds the canonical six-tool list via `buildToolList(ctx)` in stable order. Adding a 7th tool is one new file plus one registration line. See ADR-0011 for the rationale.
- **prereqs** = the shared prerequisite-check lib in `src/lib/prereqs.ts` (`checkNode`, `ensureGh`, `ensureGhAuth`, `ensureGitConfig`, `requireSetup`). The single source of truth used by both `setup` (interactive) and `init`'s [1/6] phase, so the two paths cannot drift. `ensureGhAuth` accepts an optional `scopes` list — pass `['repo', 'workflow']` from setup, omit from init's preflight.
- **requireSetup** = the check-only bootstrap gate in `src/lib/prereqs.ts`, wired into `bin/vaultkit.ts:wrap()` to run before every command except those in `SETUP_BYPASS` (`setup`, `doctor`). Throws `VaultkitError('SETUP_REQUIRED')` if Node < 22, `gh` is missing/unauthed, `gh` token lacks `repo` or `workflow` scope, or `git config user.name`/`user.email` is empty. Idempotent and fast on a configured machine — three execa calls, no network, no prompts. `claude` is intentionally NOT required by the gate (per-command commands handle its absence via `findOrInstallClaude`). The architecture fitness function in `tests/architecture.test.ts` enforces that every `src/commands/*.ts` file declares its gate posture in `tests/bootstrap-gate.test.ts` (either `COMMANDS_THAT_MUST_BE_GATED` or `BYPASS`); a new command added via `/add-command` cannot ship without addressing the gate.
- **vault layout files** = the canonical file/directory names that make up a vaultkit vault (`.mcp-start.js`, `CLAUDE.md`, `README.md`, `index.md`, `log.md`, `.gitignore`, `.gitattributes`, `_vault.json`, `.obsidian`, `raw/`, `wiki/`, `.github/workflows/`). Defined in `src/lib/constants.ts` as `VAULT_FILES`, `VAULT_DIRS`, `WORKFLOW_FILES`. Use those constants instead of inline string literals.
- **vault layout** = the operations that build / detect / repair the canonical vault file structure, in `src/lib/vault-layout.ts`. `CANONICAL_LAYOUT_FILES` lists the files in deterministic order; `detectLayoutGaps(vaultDir)` returns the missing subset; `writeLayoutFiles(vaultDir, ctx, files)` writes the named files with content from `renderLayoutFile`. Single source of truth for both `init` (full creation) and `update` (gap repair).
- **Vault class** = the rich object form of a registered vault, defined in `src/lib/vault.ts`. Wraps name + dir + expectedHash and exposes disk/path checks (`existsOnDisk`, `isVaultLike`, `hasGitRepo`, `hasLauncher`, `sha256OfLauncher`). Construct via `Vault.tryFromName(name, cfgPath?)` (returns null if unregistered), `Vault.requireFromName(name, cfgPath?)` (throws `VaultkitError('NOT_REGISTERED')` if unregistered — preferred for commands with no meaningful unregistered branch), or `Vault.fromRecord(record)`.
- **vault templates** = the static-content builders in `src/lib/vault-templates.ts` (`renderClaudeMd`, `renderReadme`, `renderIndexMd`, `renderLogMd`, `renderGitignore`, `renderGitattributes`, `renderDuplicateCheckYaml`, `renderVaultJson`). Used by `init` and `update` to scaffold or restore the standard vault layout. Separate from `vault.ts` because they are content, not logic.
- **MCP helpers** = the `claude mcp` CLI wrappers in `src/lib/mcp/mcp.ts`: `runMcpAdd`, `runMcpRemove`, `runMcpRepin`, `manualMcpAddCommand`, `manualMcpRemoveCommand`, `manualMcpRepinCommands`, `findOrInstallClaude`. `runMcpAdd` is the single source of truth for the security-critical `--expected-sha256=<hash>` argv shape — every registration flow must go through it. `runMcpRemove` is the matching invariant for removal: `destroy`, `disconnect`, and the `init` rollback path all share one argv shape, so a future `--scope` change is a single edit.
- **Logger** = the structured logging interface in `src/lib/logger.ts` with four levels (`info` / `warn` / `error` / `debug`). `ConsoleLogger` is the production implementation; `SilentLogger` is the test no-op. Replaced the flat `LogFn` type in v2.1.0. Commands receive one via `RunOptions.log`.
- **VaultkitError** = the categorized error class in `src/lib/errors.ts`. Carries a `VaultkitErrorCode` (`INVALID_NAME`, `NOT_REGISTERED`, `ALREADY_REGISTERED`, `NOT_VAULT_LIKE`, `HASH_MISMATCH`, `AUTH_REQUIRED`, `PERMISSION_DENIED`, `TOOL_MISSING`, `NETWORK_TIMEOUT`, `UNRECOGNIZED_INPUT`, `PARTIAL_FAILURE`, `RATE_LIMITED`, `SETUP_REQUIRED`) that `bin/vaultkit.ts:wrap()` maps to distinct process exit codes (2-14) via the `EXIT_CODES` table. Use `isVaultkitError(err)` for narrowing in `catch` blocks. Plain `Error` continues to exit 1 for unexpected failures.
- **CommandModule** = the lifecycle interface in `src/types.ts` that every `src/commands/<name>.ts` exports satisfies. Generic over `[TParams...]` (positional args as a tuple), `TOptions` (extends `RunOptions`), and `TResult`. Each command file ends with a `_module: CommandModule<...> = { run }` type sentinel that fails the type check if a future command's `run` signature drifts.
- **exit codes** = the public process-exit contract documented in [src/lib/errors.ts](../../src/lib/errors.ts): 0 = success, 1 = unknown error, 2-14 = `VaultkitErrorCode` categories (13 = `RATE_LIMITED` added in 2.7.1; 14 = `SETUP_REQUIRED` added by the bootstrap gate). Scripted callers may rely on these specific codes.
- **gh failure classification** = the four-way categorization in `src/lib/github/gh-retry.ts:_classifyGhFailure(status, body, stderr, headers)` that drives `ghJson`'s retry behavior: `transient` (5xx / 429 / `previous visibility change is still in progress` 422 / network reset/timeout — retry 1s/2s/4s); `rate_limited` (`secondary rate limit` / `abuse detection` / HTTP 429 — wait `Retry-After` or 60s, retry up to 3x more, then throw `VaultkitError('RATE_LIMITED')`); `auth_flagged` (`Repository '<x>' is disabled.` / `Please ask the owner to check their account.` — throw `VaultkitError('AUTH_REQUIRED')` immediately, no retry); `fatal` (everything else — throw immediately). The same `auth_flagged` text pattern is also recognized in `src/lib/git.ts` (`pushNewRepo`, `pushOrPr`) so the failure mode surfaces identically across the gh-API and git-push paths. Pure helper `_parseGhIncludeOutput(raw)` parses the status line + headers + body returned by `gh api --include` so the retry layer can read `X-RateLimit-Remaining` / `X-RateLimit-Reset` / `Retry-After`.
- **refresh** = the `vaultkit refresh [name]` command (`src/commands/refresh.ts`) — walks `raw/`, classifies each markdown source by the URL in its frontmatter, runs the appropriate check (GitHub commit-since-clip via `gh api`; non-git via `compareSource`; missing-URL routes to its own report section), writes a dated freshness report under `wiki/_freshness/<YYYY-MM-DD>.md` only when there are findings. Accepts `--vault-dir <path>` to bypass the registry for CI use.
- **freshness report** = a markdown document at `wiki/_freshness/<YYYY-MM-DD>.md` produced by `vaultkit refresh`. Has up to four sections — auto-checked git changes, auto-checked text-only-compare drift, sources couldn't auto-check (manual review), sources without a URL — and a footer pointing at the patch-flow guidance in CLAUDE.md. Skipped entirely when there are no findings (no PR noise on quiet weeks).
- **synthesis policy** = the vault-specific guidance for *how* the wiki is written and how it gets patched on refresh. Lives in CLAUDE.md's "Wiki Style & Refresh Policy" section, wrapped in `<!-- vaultkit:wiki-style:start/end -->` markers. Carries voice/structure conventions, the patch-flow constraint, the WebFetch handoff for non-git sources, and recommended Claude Code settings (model, thinking, effort) for collaborator convergence.
- **managed section** = a region of CLAUDE.md (or any future markdown file vaultkit owns) wrapped in `<!-- vaultkit:<id>:start -->` ... `<!-- vaultkit:<id>:end -->` HTML-comment markers. `src/lib/claude-md-merge.ts` provides `renderManagedSection(id, body)` and `mergeManagedSection(existingMd, id, body, headingName)` so future template updates can replace just the managed region without disturbing user edits.
- **text-compare** = the non-git source freshness check helper in `src/lib/text-compare.ts`. `compareSource(url, localMarkdown)` HTTP-fetches the URL, runs Mozilla Readability + `jsdom` to extract the article body, strips both sides to plain text, and returns Jaccard similarity. Format-noise tolerant: a Web-Clipper-formatted local clip and a fresh fetch score ~1 if their words match. Routes paywalls / SPAs / 4xx / 5xx to an `unfetchable` outcome with a reason string.

Run `/common-ground` at the start of each session to surface assumptions about this domain.
